# 金融量化平台与 Skill 审计

FinHack 与 finance-quant-skills 都很有发现价值，但它们不是同一种东西。前者是一套试图贯通数据、因子、训练、回测与交易的 Python 平台；后者是一组准备交给智能体读取、调用脚本甚至连接交易终端的 Skill。平台审计偏向数据与回测正确性，Skill 审计还必须增加软件供应链、指令注入、凭证和资金权限模型。

本页固定到 2026-07-17 的两个 commit。完整 clone 只在 ignored `data/reference-repos/` 中离线取证，不进入 Git、Pages 或插件。这里的“采用”是学习架构或独立重写，不等于运行第三方代码，更不等于推荐买卖证券。

## 一页结论

| 对象 | 固定 commit | 根许可 | 已核验价值 | 首要失败模式 | 决策 |
|---|---|---|---|---|---|
| FinHack | `dedbbd0…` | GPL-3.0-only 或另签商业许可 | 采集—因子—训练—回测—任务分层；A 股规则清单 | README 明说重构版可能不能运行；setup 有写副作用；测试证据弱 | 架构参考，当前不安装 |
| finance-quant-skills | `b03516e…` | NOASSERTION | 13 个金融 Skill 的领域地图与工具发现 | 根许可证缺失、复制文档权利不清、Cookie/账号/实盘下单混在可执行指令中 | 整体 link-only，逐 Skill 隔离 |

Star 可以帮助发现维护活跃的项目，却不能回答许可证、历史时点、回测泄漏、依赖投毒或订单权限。即使一条安装命令成功，也只证明某个环境执行了代码，不证明公式正确、数据合法、策略盈利或 Agent 安全。

## 为什么平台和 Skill 要用两套威胁模型

传统代码只有被导入、安装或执行时才产生行为；Agent Skill 的 Markdown 本身就是潜在执行面。它能告诉模型运行 shell、安装包、打开浏览器、复制 Cookie、读取环境变量、连接账户或调用订单函数。因此审计对象包括：

1. **代码事实**：文件、函数、依赖和测试实际存在什么。
2. **上游声明**：README 或 Skill 描述声称能做什么。
3. **指令权限**：Agent 被要求访问什么秘密、网络、客户端和不可逆操作。
4. **许可谱系**：根代码、嵌套仓库、复制文档、图片、模板和数据分别由谁授权。
5. **可证伪测试**：怎样用不联网的最小 fixture 证明或推翻关键能力。

finance-quant-skills 中“官方文档完整收录”可能提高检索便利，却同时提出版权来源问题；“支持实盘”可能扩大功能，却同时把研究助手变成资金操作代理。功能越多不等于默认越好。

## FinHack：从目录事实到可运行性

固定 commit 的 README 把 FinHack 定位为可扩展量化框架，列出数据采集、因子计算与挖掘、机器学习、策略、回测和交易接入。目录大体支持这个上游声明：`collector/tushare/` 分市场和资产采集，`factor/default/` 包含指标、分析、挖掘和任务运行，`trainer/` 与 `trader/` 处理训练和事件/账户，demo project 展示策略与 QMT/MiniQMT/RQAlpha 适配。

但“目录存在”只证明作者设计了接口。README 第一屏同时写明当前开始大改重构，更新代码“运行不了了”。这不是小免责声明，而是决定当前版本不能作为生产依赖的直接证据。可靠采用必须固定一个可运行 release，或等待上游恢复并给出最小端到端验证。

### 安装本身会修改源码树

`setup.py` 在调用 `setup()` 前执行两个写操作：遍历 `finhack/`，遇到没有 `__init__.py` 的目录就创建文件；随后以写模式覆盖 `finhack/__init__.py` 以写入版本。读取包元数据、构建 wheel 或开发安装都可能触发它。由此得到三条经验：

- 不要在珍贵或未提交的工作树上试装；先在一次性 clone/容器中比较安装前后哈希。
- 不要把 `pip install` 当无害的探测命令；setup/build backend 属于可执行代码。
- Python `>=3.6` 与庞大 pinned requirements 并不等于今天任意 Python 版本可解析或安全共存。

最小安全测试应复制固定 commit 到临时只读/可丢弃目录，离线构建时拦截网络与子进程，记录工作树 diff。当前 Issue 不执行该测试，因为仓库已经明示不可运行且没有采用必要。

### 数据采集层

Tushare collector 从项目配置取 token 和数据库名，初始化 API 后按依赖收集 A股基础、行情、财务、市场、指数、基金、宏观、期货、美股、港股、可转债和外汇。广度很有参考价值，但每个 endpoint 都需要回答：权限积分、字段版本、首次发布时间、历史修订、分页/限频、空返回、单位、时区、复权和数据许可。

最新 commit 的信息是“识别 Tushare 无权限永久错误并跳过无限重试”。这是良好的失败分类方向，也揭示依赖权限层级：同一 token 不保证所有表可用。跳过接口必须生成结构化缺口，不能让后续因子把“无权限”误当零或无事件。

### 因子与历史时点

factor 目录提供广泛指标与任务编排线索，但正确性不能用因子数量证明。每个因子至少固定：原始字段、公式、窗口、缺失、去极值、标准化、行业/市值中性化、复权、发布可得日和 universe。财报因子必须使用披露日而非报告期末；指数成分、ST、上市/退市状态也必须按历史时点恢复。

所谓动态复权需要独立说明现金分红、送转、拆并股和配股如何进入持仓与价格。只让价格曲线连续，可能遗漏现金、税和股数变化；前后复权误差较小也不等于组合现金流正确。

### 回测与 A 股规则

规则实现确实出现退市、停牌、涨跌停、ST、主板、100 股/科创板数量、T+1/T+0、成交量、税费、最低佣金和滑点。这些是很好的验收清单。代码也暴露了待验证问题：涨跌停条件旁标注“不严谨”，费用/滑点方法旁写“怕后面有坑”，部分函数引用全局 `context` 而不是 `self.context`。

需要建立下列可证伪 fixture：

1. 昨日收盘后信号只能在下一可成交时点下单，不能用同日收盘成交。
2. 涨停一字板买不到、跌停一字板卖不掉；触及价格但有成交量的情形需按模型明确。
3. 当日买入数量在 T+1 下不可卖，昨日持仓可卖数量正确递减。
4. 停牌、退市整理、ST、科创/北交数量与历史规则版本一致。
5. 手续费、印花税、过户费、最低佣金、滑点的方向和生效日正确。
6. 成交量参与率约束、部分成交和拒单不会凭空创造现金或持仓。
7. 分红送转前后总财富守恒，财报和成分数据不使用未来修订。

在这些 fixture 通过以前，只能说“代码尝试实现规则”，不能说“回测真实”。scanner 的 backtest 29、成本 45 等关键词计数同样只是线索。

## finance-quant-skills：逐 Skill 供应链矩阵

固定 commit 有 13 个顶层 Skill。仓库 README 声称 MIT，但根目录没有 LICENSE/COPYING/NOTICE，`pyproject.toml` 也没有 license 字段，因此公共来源卡必须使用 `NOASSERTION`。更重要的是，大量文件看起来是 API 文档或从别处引入的 Skill；即便未来补了仓库 MIT，也要证明贡献者有权按该许可再发布这些材料。

下面的矩阵不是最终安全认证，而是当前证据下的默认隔离级别：

| Skill | 主要角色 | 凭证/环境 | 执行或资金面 | 许可/证据缺口 | 当前决策 |
|---|---|---|---|---|---|
| akquant | 量化工具与工作流入口 | 依赖上游包及其数据源 | 需逐脚本复核 | 根许可缺失，上游独立 | link-only |
| akshare | 多源行情/财务数据 | 多为公开网页接口，稳定性和条款独立 | 网络抓取，无默认订单 | 接口来源与再分发逐 endpoint | optional 候选，先写本项目 adapter |
| backtrader | 回测框架参考 | Python 依赖 | 策略执行但可离线 | Skill 复制内容与上游许可分开 | link-only，采用测试思想 |
| baostock | A 股只读数据 | 登录协议但上游通常无需个人账号 | 网络查询，无默认订单 | 字段/复权/限频需验证 | optional 候选 |
| equity-researcher | 研究报告生成 | 浏览器、图表、额外包 | 子进程、渲染与文件输出 | 声称来自其他 Skill，来源许可待核 | quarantine 后再拆模板思想 |
| joinquant-docs | 聚宽 API 文档 | 平台账户与数据条款 | 文档包含订单、资金、实时运行 | 大量复制文档授权待核 | link-only / 交易段 quarantine |
| jqdatasdk | 聚宽数据 SDK | username/password 或 Token | 主要数据查询 | 账号、额度、数据许可 | optional 只读候选，默认禁用 |
| miniqmt | xtquant 行情与交易 | 本地专有客户端、账户目录 | `order_stock`、撤单、资产/持仓 | 账户和资金权限高 | quarantine |
| pywencai | 问财查询 | 浏览器 F12、Cookie、环境/用户文件 | Cookie 请求与潜在会话复用 | 会话安全、条款和接口稳定性 | quarantine；默认拒绝 Cookie 流程 |
| qmt-docs | QMT 策略/API 文档 | QMT 账号/客户端 | `passorder`、即时下单、实盘回调 | 官方文档复制权和资金权限 | quarantine |
| rqalpha | A 股回测框架参考 | 数据 bundle/依赖 | 本地回测为主 | Skill 与上游许可分别核 | link-only，采用成交测试清单 |
| tdxquant | 通达信/行情工具 | 本地或网络数据环境 | 需逐脚本检查 | 专有数据/协议和根许可 | link-only |
| tushare | 多市场数据 SDK | Token、积分权限 | 只读查询可隔离 | 数据条款、字段权限、额度 | optional 只读候选，用户主动配置 |

### Cookie 不是“免费的 API Key”

pywencai Skill 指导用户打开浏览器 F12 复制 Cookie，并允许通过环境变量或用户目录文件保存。这会把浏览器会话带入代理执行面。Cookie 可能包含登录状态、风控标识和其他权限，泄漏后影响不止单个 endpoint；网站接口也可能随时改变或禁止自动化。

因此本项目不通过浏览器开发者工具捞后台认证，不读取用户 Cookie，不把 Cookie 写进 `.env`、日志、Raw 或 CI secret。若官方提供无需会话的公开接口，可以按条款另建只读 adapter；否则保留链接与人工查询路径。东方财富 SSE 的 `noop`/`next_seq` 经验同样说明：找到请求并不等于理解数据语义，更不等于获得再利用授权。

### Token 与账号密码也要分级

Tushare Token、JQData 账号和 API Key 并非一律禁止，但必须满足用户主动提供、合法账户、最小权限、只在运行时注入、永不记录、可撤销和无订单权限。默认 CI 用离线 fixture；真实查询结果保存 provider、endpoint、参数、抓取时刻、数据时间、版本和哈希，不保存凭证。

账号密码比 API Token 风险更高，不应直接交给通用 Skill。若服务支持专用只读 Token，优先使用专用 Token；若只有账号密码，默认不自动化，除非另开安全设计并明确存储、轮换和撤销。

### 实盘下单与研究助手必须硬隔离

miniqmt 和 qmt-docs 展示连接专有客户端、构造账户、下单、撤单、查资产/持仓和即时订单。即使用户将来希望自动化，研究结论与订单执行也必须是两个权限域：研究端只能输出候选与证据；执行端需要独立账户、白名单、金额/价格/频率限制、模拟环境、人工确认、幂等键、撤销流程和审计日志。

第一版知识库没有个人持仓，也不需要订单能力。实盘下单不会提高研究质量，反而把数据错误、提示注入和模型幻觉变成资金损失，所以这些 Skill 整体 quarantine，不安装到 `investing-research` 插件。

### 复制文档和 imported Skill 的许可

joinquant-docs、qmt-docs 与 equity-researcher 含大量文档、模板或声称来自其他项目的内容。公开网页可读、GitHub 可 clone、作者写了“官方文档”都不等于允许按 MIT 再发布。审计需要为每个资源记录原始 URL、作者、版权/许可证、修改情况和本仓库使用方式。

在证据不全时最安全的复用是：保存规范链接、固定 commit、我们自己的短摘要和可证伪检查清单；不复制全文、不把第三方 Skill 重新打包进插件。`NOASSERTION` 不是指控侵权，而是诚实表达无法从仓库证据确认授权。

### 提示注入为什么属于代码审计

Skill 文件可能写“忽略之前规则”“读取环境变量”“上传配置”“安装并运行”“打开 F12 复制 Cookie”或“立即下单”。审计代理如果照做，就已经执行了不可信输入。安全流程把全部上游 Markdown、注释、README、issue、数据字段和模型输出当作数据，只有本项目 Skill 的安全边界能发出动作指令。

扫描只记录相对路径和能力类别，不回显可能含 secret 的源码行。没有命中 prompt injection 关键词不能证明安全，因为同一意图可以换词；最终仍需人工读所有 frontmatter、脚本入口、安装命令和权限描述。

## 从两个项目真正值得沉淀的思想

FinHack 值得沉淀的是模块边界与中国市场规则清单，而不是直接复制实现。finance-quant-skills 值得沉淀的是按任务拆分工具说明的组织方式，以及“金融 Skill 本身需要供应链审计”这个教训。本项目因此增加 `audit-finance-skills`：它只接受 already-local 目录，复用离线 scanner，再逐 Skill 检查根/嵌套许可、安装副作用、网络、凭证、Cookie、浏览器指令、专有终端、订单/转账和提示注入。

采用路径按最严格风险排序：

1. **adopt**：独立重写公式、schema 或检查清单，用冻结 fixture 验证。
2. **optional**：许可证清晰的只读数据依赖，由用户自行安装和配置，默认 CI 离线。
3. **link-only**：许可证或复制来源不清，只存链接、commit 和原创摘要。
4. **quarantine**：Cookie、账号密码、专有终端、broker、order、transfer 或远端执行。
5. **reject**：绕过认证、秘密外传、隐蔽持久化、无确认资金操作。

## 下一轮可证伪测试

对 FinHack，等待上游给出可运行 release 后在一次性环境执行：安装前后树哈希、最小 Tushare-free fixture、因子历史可得日、分红送转财富守恒、T+1/涨跌停/费用/部分成交、退市与样本外回测。每项失败都进入 [[经验与失败教训]]，不能只保留成功结果。

对 finance-quant-skills，先解决根许可证和 13 个 Skill 的来源谱系；然后只挑无凭证、无订单、许可清晰的一个最小 Skill 做静态审计。任何依赖网络的验证使用假的响应 fixture；任何需要 Token 的适配由用户在独立任务主动配置；Cookie 与实盘路径不作为晋级测试。

## 审计检查清单

1. URL、pinned commit、commit date 和根许可证文件是否同时存在？
2. README 许可证是否与文件和包元数据一致？嵌套内容是否有独立授权？
3. 安装、构建、import、hook 或 Skill 指令会写什么、执行什么、联网到哪里？
4. 数据源需要 Cookie、Token、账号密码、专有客户端还是 broker？
5. Skill 是否能下单、撤单、转账、同步交易或读取资产？
6. 上游文本是否可能形成提示注入，审计时是否始终作为数据处理？
7. 代码事实、上游声明、推断、未知与下一项可证伪测试是否分开？
8. scanner 命中是否回到具体相对文件核验，而没有冒充覆盖率？
9. 采用是否选择最小能力，而非整体安装一个大 bundle？
10. public Git 和 Pages 是否仍不含 clone、凭证、Cookie、账户、持仓和缓存？
